# duyi-resume-writer 完整工作流程

执行以下 8 个阶段。阶段中的硬规则详见对应 references 文件。

## Phase 1：读取与解析

通读原始简历，标记：
- 公司名称、职位、时间段
- 每段经历中的具体动作
- 所有能提取的量化数据
- 明显的时间错误（未来时间、经历重叠等）

## Phase 2：时间校准

详见 [timeline-rules.md](timeline-rules.md)。

核心要求：
- 所有经历结束时间必须在今天之前
- 工作经历串行、无重叠、无空窗
- 每段经历 ≥7 个月
- 大学期间项目可与学业并行

## Phase 3：项目制重构

**拒绝写岗位职责**，全部重写为项目经历。

每个项目必须包含：
- 项目名称
- 负责事项（具体动作：选题 / 拍摄 / 剪辑 / 私域 / 投放 / 直播等）
- 项目结果（横向排列的数据，优先用绝对值）

项目分类：
1. **内容增长类**（涨粉 / 播放 / 爆款数）
2. **私域 / 用户类**（社群 / 复购 / 用户生命周期）
3. **商业 / 直播类**（GMV / 挂车 / 场观）
4. **投放 / 商务类**（ROI / BD / 投流金额）

项目编号规则：无论项目属于同一家公司还是不同公司，统一按「项目 1、项目 2、项目 3……」排列。

## Phase 3.5：跨行业经历强制重构

**只要用户提供了任何工作经历，就必须从经历中挖掘可迁移到新媒体运营的能力，强制重构成新媒体运营项目。**

详见 [cross-industry-migration.md](cross-industry-migration.md)。

## Phase 4：数据补全

- 原始数据能用则用，**去百分比**，改为绝对值
- 数据缺失时，根据岗位常见产出合理推断：
  - 视频产出：30 条/月（全职）、10 条/月（兼职/实习）
  - C 端获客：100-200 人/月
  - 社群规模：按行业常见值估算
- 所有补全数据必须标注：`【数据为 AI 推断，需你确认】`

## Phase 4.5：数据质检与合理性修正

详见 [data-qc-rules.md](data-qc-rules.md)。

核心要求：
- 不允许把账号/团队历史累计算到个人头上
- 年产内容 ≤600 条
- 留资/引流数据必须按行业基准核查
- 变现金额区分个人业绩与公司 GMV

## Phase 5：个人优势写作

公式：**技能定义 + 累计总数据支撑**

强制约束：
- 每条个人优势必须有量化数据
- 所有数据必须是下方项目经历数据的**严格总和**，计算公式：
  ```
  优势数据 = 项目1对应数据 + 项目2对应数据 + ... + 项目N对应数据
  ```
- **禁止去重、禁止估算、禁止写「约 X 条」与明细对不上**。如果项目经历明细是 800+400+50=1250，优势必须写 1200+ 或 1250+，不能写 500+。
- 若无法闭环，标注【数据待补齐】
- 写完个人优势后，必须反向核对：把每个数据拆解回项目经历，看是否能一一对应

## Phase 6：生成工作讲述

详见 [oral-story-template.md](oral-story-template.md)。

生成两个版本：
1. 面试口述版（800-1200 字）
2. 个人经历详细版（不限制字数）

## Phase 7：缺失经历标注

如果 AI 补充了简历中原本没有的经历或数据，在输出末尾列出：

```markdown
## AI 补充说明

以下内容为 AI 基于岗位常见情况补充，简历上原本缺失，请根据实际情况修改或删除：

1. **项目 X：XXX** — 补充原因：[说明]
2. **数据：XXX** — 原简历未提供，AI 按行业均值推断
3. **场景化细节：XXX** — 工作讲述中的合理演绎，需用户面试前核实
4. **时间重叠：XXX** — 原经历存在重叠，已按并行/主次方式处理
```

## Phase 8：导出 docx

详见 [output-format.md](output-format.md)。

强制输出两个 docx 文件到 `~/Desktop/`：
1. `{姓名}-{求职意向}-简历-修正版.docx`
2. `{姓名}-{求职意向}-面试讲述.docx`

使用脚本：
- `scripts/generate_resume.py`
- `scripts/generate_interview.py`

模板位置：`assets/template.docx`
